How I Built a Streaming Voice AI Pipeline End-to-End

Real-time speech recognition, LLM inference, and text-to-speech with interruption support and ultra low latency.

What I Was Building

I wanted to add a voice interface to my portfolio site as a useful way for recruiters or collaborators to ask questions about my work without reading walls of text.

The user experience goals were specific:

  • Natural conversation flow—no "press to talk" buttons, no awkward silences while waiting for responses
  • Interruption support—the user should be able to ask a new question mid-response
  • Low perceived latency—audio should start playing within ~1 second of the user finishing their question

The constraint that shaped everything: my site runs on Vercel serverless functions, which don't support WebSockets. This ruled out the obvious architecture of routing all audio through my server.

Architecture Overview

Here's what I ended up with:

┌─────────────────────────────────────────────────────────────────┐
│                         BROWSER                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐         │
│  │   Mic       │───▶│  STT WS     │───▶│  Transcript │         │
│  │   Capture   │    │  (Deepgram) │    │             │         │
│  └─────────────┘    └─────────────┘    └──────┬──────┘         │
│                                                │                 │
│                                                ▼                 │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐         │
│  │   Speaker   │◀───│  TTS WS     │◀───│  LLM Call   │◀────────│
│  │   Playback  │    │  (Deepgram) │    │  (Vercel)   │         │
│  └─────────────┘    └─────────────┘    └─────────────┘         │
└─────────────────────────────────────────────────────────────────┘

The key insight: the browser is the orchestrator. It connects directly to Deepgram's WebSocket APIs for both speech-to-text (Flux) and text-to-speech (Aura). Vercel only handles two short-lived HTTP requests:

  1. Token endpoint—mints a short-lived Deepgram API key (60s TTL) so the real key never touches the browser
  2. LLM endpoint—receives the transcript, streams back response text via a RAG-augmented pipeline

This sidesteps the "Vercel can't do WebSockets" problem entirely. The client maintains the real-time connections; the server just does quick request-response work. It also means my infrastructure costs stay near zero. Vercel handles the scaling, and I'm only paying for API calls.

Streaming Design Choices

WebSockets vs SSE

For the audio streams (STT and TTS), WebSockets were the only real option. You need bidirectional, binary-capable connections for real-time audio. SSE is text-only and unidirectional—fine for streaming LLM tokens, not for 16kHz PCM audio frames.

For the LLM call, I use a standard HTTP POST that returns a streaming response. The browser reads it chunk by chunk and pipes each text fragment to the TTS WebSocket. This works fine because text tokens are small and infrequent compared to audio frames.

Audio Format and Frame Cadence

I settled on linear16 PCM at 16kHz for both STT input and TTS output. It's the format Deepgram works with natively, and it avoids transcoding overhead.

On the capture side, the browser's MediaRecorder API doesn't give you raw PCM, so I use a ScriptProcessorNode. I pull 4096-sample buffers (~256ms at 16kHz), convert float32 to int16, and send them over the WebSocket.

Frame size is a latency-vs-overhead tradeoff. Smaller frames get audio to the STT engine faster but generate more WebSocket messages. Larger frames are more efficient but add buffering delay. I landed on 4096 samples (~256ms at 16kHz). This is small enough that the user doesn't notice capture lag, large enough that the WebSocket isn't drowning in tiny packets.

Backpressure and Buffering

The TTS stream is where buffering gets interesting. Deepgram generates audio faster than real-time playback, so chunks pile up on the client. The naive approach—playing each chunk as it arrives—produces audible gaps between segments. setTimeout and setInterval are too jittery for audio timing.

The solution was to lean on the Web Audio API's built-in scheduling. Each AudioBufferSourceNode accepts a precise start(when) timestamp on the audio clock. I track a running nextPlayTime and schedule each incoming chunk to begin exactly where the previous one ends. The browser's audio subsystem handles the rest—no gaps, no overlaps, and no reliance on JavaScript's event loop timing.

This also gave me an easy interruption hook: since every scheduled source is tracked, cancelling playback is just a loop of .stop() calls on the active sources.

Latency Budget

Here's roughly where the time goes for a typical exchange:

Stage Time
User finishes speaking → Deepgram detects end-of-turn 200-400ms
Final transcript arrives at browser 50-100ms
HTTP request to Vercel LLM endpoint 100-200ms
LLM generates first tokens 300-500ms
TTS WebSocket connection + first audio chunk 200-300ms
Total: user stops talking → audio starts ~1-1.5s

This is perceptibly fast enough to feel like a conversation rather than a query-response system. The streaming helps a lot because you hear the first word before the LLM has finished generating the full response.

Interruptions and Barge-In

This was the trickiest part to get right. The goal: if the user starts talking while the AI is still speaking, immediately stop the AI and listen to the user.

Detection

Deepgram Flux sends a StartOfTurn event when it detects the user has begun speaking. I listen for this event even while TTS audio is playing.

Cancellation Semantics

When an interruption is detected, I cancel everything in flight:

  1. Stop all scheduled audio—call .stop() on every queued AudioBufferSourceNode
  2. Close the TTS WebSocket—no point receiving more audio we won't play
  3. Abort the LLM fetch—use an AbortController to cancel the in-flight HTTP request
  4. Clear the audio queue—discard any buffered chunks
  5. Update UI state—switch waveform from "speaking" to "listening"

All of this happens synchronously in the StartOfTurn handler. The latency from "user starts talking" to "AI shuts up" is typically under 100ms.

Avoiding "Talking Over the User"

The other failure mode is the AI starting to respond while the user is still mid-sentence. Deepgram Flux handles this with turn detection. It doesn't fire EndOfTurn until there's a natural pause.

I also added a flag (isProcessingResponse) that prevents new LLM calls from starting if one is already in flight. This avoids race conditions where a partial transcript triggers a response that gets interrupted by the final transcript.

Token Security and Lifecycle

Letting the browser connect directly to Deepgram means the browser needs an API key. Shipping the real key to the client is obviously a non-starter.

The solution: server-minted ephemeral keys. When the user starts a call, the client hits a token endpoint on my server, which calls Deepgram's key creation API and returns a temporary key scoped to usage:write with a 60-second TTL. The real key never leaves the server. If someone intercepts the ephemeral key, it's useless in under a minute and it can't read data or manage the account.

The problem I didn't anticipate: 60 seconds is shorter than a conversation. The STT WebSocket stays alive because it was authenticated on connect, but TTS connections are opened per response. If the user asks a second question after the token expires, the new TTS connection silently fails. Deepgram just rejects the handshake with no useful error message. From the user's perspective, the AI just stops talking.

I solved this with client-side token rotation. A timer fires at the 50-second mark, about 10 seconds before expiry, and fetches a fresh token in the background. The new token is swapped into the variable that TTS connections read from. Existing connections are unaffected (they already authenticated), and the next TTS connection picks up the fresh key automatically. The rotation repeats for the lifetime of the call.

The tradeoff that shaped the TTL choice: shorter tokens are more secure but require more frequent rotation, which burns through the rate limit on the token endpoint (5 mints/minute). At 60 seconds with a 50-second refresh, I'm at ~1.2 requests/minute—well under the limit with headroom for retries.

What Broke Under Load

Deepgram Rate Limits (429 Errors)

The biggest issue. Deepgram has concurrency limits on TTS WebSocket connections. Even with a single user, rapid-fire questions could exhaust the quota if previous connections weren't properly closed.

Fix: Explicit connection lifecycle management. I track active connections, enforce a minimum interval between new connections (2 seconds), and always call close(1000) with a clean status code before opening a new one. I also added retry logic with exponential backoff (1s, 2s, 4s, 8s) for transient failures.

Audio Artifacts on Interruption

Early versions would sometimes play a fraction of a syllable after interruption. The issue was that AudioBufferSourceNode.stop() doesn't take effect until the next audio processing quantum (~3ms).

Fix: Call stop() with an explicit time slightly in the future (audioContext.currentTime + 0.01), which ensures the stop is processed cleanly.

Memory Leaks in Long Sessions

The audio queue could grow unbounded if chunks arrived faster than they were played. Not a problem for short interactions, but noticeable in 10+ minute sessions.

Fix: Prune the queue when chunks complete playback. Each AudioBufferSourceNode has an onended callback which is used to remove the chunk from tracking.

What I'd Improve Next

The current implementation works, but there's room to grow:

  • AudioWorklet migration—ScriptProcessorNode runs on the main thread and can cause audio glitches during heavy JS execution. AudioWorklet runs in a separate thread.
  • Opus encoding—sending raw PCM over WebSocket is bandwidth-inefficient. Opus would cut bandwidth by ~10x with minimal latency cost.
  • Voice activity detection on the client—right now I send all audio to Deepgram, even silence. Client-side VAD could reduce costs and server load.
  • Speculative TTS—start generating audio for the first few tokens before the full LLM response is ready. This could shave another 200-300ms off perceived latency.
  • Graceful degradation—if Deepgram goes down mid-call, the system should fall back to text chat rather than failing silently
